API

Conception et consommation de services Web

(Note: il sera question ici uniquement d’API REST, pas d’API SOAP)

Bien que ça ne soit pas une technologie à proprement parler, le travail dans un contexte technique incluant une API est un incontournable dans le milieu professionnel. La taille des écosystèmes et des applications en général oblige à séparer les projets, sans même parler de la diversité technologique.

Une API va permettre à des applications de communiquer entre elles, ou plus généralement à une application de s’ouvrir à une autre. Elle s’ouvre via ce que l’on appelle des “endpoints” qui sont des url que l’on peut appeler de l’extérieur. On peut configurer un endpoint pour qu’il soit appelable avec une certaine méthode HTTP, on utilise fréquemment les méthodes GET, POST, PUT et DELETE.

Savoir travailler avec une API, c’est plus globalement savoir faire communiquer les applications entre elles, et comprendre les notions de réseau basiques dont on a besoin pour faire ça. J’ai eu l’occasion de travailler avec beaucoup d’API différentes au cours de mon expérience en alternance et durant mes projets à l’école. C’est selon moi un incontournable du développement applicatif de nos jours.

Anecdotes :

Comme dit au dessus, j’ai travaillé avec des API tout au long de mon parcours depuis mon introduction à ce concept en 2ème année à l’ESIEA. Pour n’en citer que deux je choisirais l’API du projet neoherba et l’API des outils internes de mon entreprise.

Concernant le projet neoherba, comme détaillé dans les réalisations, cela a été ma première expérience dans le développement d’une API remplissant des fonctions de CRUD et dans la communication avec une API dans un site web. J’ai mis en pratique cette compétence en développant des services permettant la gestion des données du côté de l’API, et en appelant ces services dans les composants frontend.

Le projet de refonte des outils internes quant à lui a été pour moi une occasion de plus travailler autour de la conception d’une architecture cohérente pour une API plus volumineuse. En effet le but était d’unifier toutes les API déjà existantes pour les outils en une seule, le but étant de centraliser dans une boîte à outils des features assez similaires et aussi de rendre plus homogène la stack technique (plus de détails à ce sujet dans la page de la réalisation). Cette démarche de centralisation a aussi permis de protéger tous les outils derrière une même authentification keycloak, ce qui n’était pas le cas avant.

Autocritique :

J’ai eu l’occasion de travailler dans de nombreux écosystèmes logiciels comportant des API au cours de mon parcours, et c’est donc assez naturellement que je pense avoir un niveau confirmé dans cette compétence. J’ai eu l’occasion de travailler à la fois dans le développement d’API et dans la communication avec des API, ce qui me rend compétent des deux côtés du spectre.

Il me reste encore des choses à apprendre d’un point de vue de la mise en pratique de certains de mes savoirs théoriques. Je n’ai par exemple jamais eu l’occasion de mettre en pratique une vraie structure entièrement Restful, car c’est quelque chose que l’on a rarement l’occasion de faire dans un contexte d’entreprise à cause du temps que cela consomme. Je souhaite approfondir cet aspect dans un projet personnel, sûrement en allant de pair avec mon approfondissement de React.
